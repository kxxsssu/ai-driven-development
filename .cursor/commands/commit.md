# 커밋 메시지 작성

현재 스테이징/변경 사항을 분석해 Conventional Commits 형식의 커밋 메시지를 작성하고 커밋한다.

## 진행 방식
1. `git status`와 `git diff`로 변경 내용을 파악한다.
2. 변경의 성격을 판단한다: `feat`(신규 기능), `fix`(버그 수정), `refactor`, `style`, `docs`, `test`, `chore`.
3. 다음 형식으로 메시지를 작성한다.

```
<type>(<scope>): <한 줄 요약>

- 변경 이유와 핵심 내용 (선택)
```

## 규칙
- 요약은 한국어로 간결하게(50자 이내 권장), "무엇을 왜" 중심으로 작성한다.
- scope는 변경 영역(예: `api`, `gallery`, `auth`)을 쓰되 불명확하면 생략한다.
- 관련 없는 변경을 한 커밋에 섞지 않는다.
- `.env` 등 민감 파일이나 빌드 산출물(`.next/`)은 커밋하지 않는다.
- **사용자가 명시적으로 커밋을 요청한 경우에만** 실제 `git commit`을 실행한다.
